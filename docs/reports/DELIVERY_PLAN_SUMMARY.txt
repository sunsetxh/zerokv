================================================================================
TOP 3 TECHNICAL RISKS & MITIGATION
================================================================================

RISK 1: RDMA Implementation Complexity (Probability: 15–20%, Impact: HIGH)
  What Could Go Wrong:
    UCX RMA API incomplete or undocumented
    RDMA put/get implementation harder than expected
  
  Fallback Strategy:
    Week 6–7: Spike research on UCX RMA API
    Week 8: Decision gate (proceed or pivot to messaging-only MVP)
    If blocked: Ship US1–US4 at Week 14; defer US3 to Phase 2
  
  Timeline Impact: +4–8 weeks if unmitigated

---

RISK 2: Memory Registration Performance (Probability: 25–30%, Impact: MEDIUM)
  What Could Go Wrong:
    Registration becomes bottleneck
    Can't achieve 90% bandwidth target for large messages
  
  Fallback Strategy:
    Week 5: Benchmark vs NCCL (establish baseline)
    Week 12: Add MemoryPool (slab allocator)
    Week 15: Add RegistrationCache (LRU)
    If needed: Relax perf target for unregistered buffers
  
  Timeline Impact: +2–4 weeks if unmitigated

---

RISK 3: Ascend NPU Plugin Integration (Probability: 35–40%, Impact: MEDIUM–HIGH)
  What Could Go Wrong:
    HCCL API incompatible with plugin design
    Ascend support breaks in Phase 2
  
  Fallback Strategy:
    Week 2–3: Design plugin interface (decouple from core)
    Week 6: Early Ascend team engagement
    Week 12–14: Implement mock HCCL plugin (validate architecture)
    Week 15: Decision gate (proceed or defer to Phase 2)
  
  Timeline Impact: +4–6 weeks if unmitigated; can shift to Phase 2

---

DECISION GATES (Weekly Risk Review):
  ✓ If probability rises above 50% OR impact is CRITICAL
    → Emergency escalation meeting
    → Re-plan with fallback strategy

  ✓ 1KB message:     < 3 microseconds

  ✓ < 5% for 64KB+ messages

Scalability:
  ✓ 64 concurrent connections without degradation

Quality:

  ✓ 70% branch coverage
  ✓ ASAN clean (no memory errors)
  ✓ TSAN clean (no data races)

WHAT'S IN MVP (6 User Stories)
================================================================================

✅ US1: Point-to-Point Tag-Matched Messaging
   - Tag send/recv with 64-bit tags
   - 0 bytes to 1GB message sizes
   - Zero data corruption
   
✅ US2: Connection Lifecycle Management
   - Listen, accept, connect, close
   - Connection timeouts
   - 64 concurrent connections
   
✅ US3: One-Sided RDMA Operations
   - Put (write to remote memory)
   - Get (read from remote memory)
   - Flush (ensure completion)
   
✅ US4: Asynchronous Operation Model
   - Future<Status> for all operations
   - Poll (ready()), block (get()/wait()), callbacks
   - Future::wait_all() for multiple operations
   
✅ US5: Memory Registration and Management
   - Register/deregister buffers
   - Pin/unpin pages with UCX
   - Obtain serializable remote keys
   
✅ US6: Library Configuration and Initialization
   - Config builder pattern
   - Environment variable overrides
   - Sensible defaults

================================================================================
WHAT'S NOT IN MVP (Deferred to Phase 2+)
================================================================================

❌ Python bindings       (Phase 2 - can parallelize)
❌ MemoryPool            (Phase 2 - optimization)
❌ RegistrationCache     (Phase 2 - optimization)
❌ Scatter/gather        (Phase 2 - advanced)
❌ NCCL plugin           (Phase 2 - requires external API stability)
❌ HCCL plugin           (Phase 2 - requires Ascend team)
❌ Multi-path RDMA       (Phase 3 - advanced)
❌ Observability (metrics/tracing) (Phase 3 - ops nice-to-have)

MVP is intentionally focused on core AXON functionality.
Phase 2 will add Python, optimization, and plugin ecosystem.
